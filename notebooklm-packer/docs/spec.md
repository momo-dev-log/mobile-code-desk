## 0. 主要決定サマリ

- 本文取得は基本1回。Workerが取得・charset正規化・status判定を行い、本文抽出はブラウザ側で実施してIndexedDBに保存する
- 検索・立ち読み・候補・ファイル生成はすべて保存済み本文を参照する
- 出力の正本は Markdown（パック全体1ファイル）。txtは任意の同時出力。PDFはv1ではやらない
- Markdown/txt生成はブラウザ側
- 取得失敗した記事はパック候補に追加できない
- 本文が短い記事（200字未満）は失敗にせず、警告付きで保存して人間が立ち読みして判断する
- sitemap取り込みは最小形で対応する（展開→件数表示→選択→取り込み、展開上限200件）
- 検索はスペース区切りの語をORで拾い、含む語が多い記事を上位に表示する。関連度は高/中/低バッジで表示する
- 除外キーワードはv1.3候補として見送る
- 複数URL取り込み時は確認を1枚挟んでから取得。1件のみなら即時取得可
- パックは30件を性能目安とし、50件超で警告（ハードリミットなし）
- 現行アプリは `notebooklm-packer/v1/` に温存して並行開発する。Workerは別名で新規デプロイし、旧Workerは現行アプリ用に残す
- UI文言に「送信」は使わない

## 1. 目的

WebサイトのURLから本文を取得し、人間が中身を確認しながら選別して、NotebookLMに入れる前の「資料パック」を作るツール。NotebookLMへの自動送信はしない。スマホのブラウザでの利用を主とする。

コンセプトは本屋方式。

- **さがす** ＝ 棚（検索結果カードを眺め、立ち読みする）
- **パック候補** ＝ カゴ（印を付けるだけ。処理は走らない）
- **仕上げ** ＝ レジ（保存済み本文からファイルを生成する）

構成はブラウザSPA＋Cloudflare Worker。Workerの責務は「HTML/XMLの取得・charset正規化・httpStatus判定・サイズ上限・タイムアウト・構造化返却」。本文抽出・XML解析はブラウザ側で行う。

### 移行方針

- 現行アプリ一式は `notebooklm-packer/v1/` へ移して温存し、新版を並行開発する
- Workerは新規ファイル・別名でデプロイする。旧Workerは現行アプリ用に残す
- Worker改修は本仕様9章対応の一回きりとし、以後は凍結する

## 2. 用語集

| 用語 | 意味 |
|---|---|
| 記事（article） | 取り込んだURL1件分のデータ。メタ情報と本文からなる |
| パック（pack） | NotebookLMに入れる資料のまとまり。記事IDの配列を持つ。v1ではdefaultパック1個のみ |
| 取り込み | URLを登録すること。この時点では取得しない |
| 本文取得 | Worker経由でHTMLを取得し、ブラウザで本文を抽出してIndexedDBに保存すること。1記事につき基本1回 |
| sitemap展開 | sitemap URLからURL一覧を取り出し、取り込み候補として表示すること |
| 立ち読み | 保存済み本文をカード上でプレビューすること（通信なし） |
| パック候補に追加 | pack.itemsに記事IDを加えること。本文取得やファイル生成は走らない |
| 仕上げ | 保存済み本文からMarkdown/txtを生成すること（通信なし） |
| 最新版を取り直す | 任意操作としての再取得。成功時のみ既存データを置き換える |
| scorer | 検索クエリと本文から関連度を計算する関数。唯一のAI差し込み口 |

**禁止用語**: 「送信」をUI文言に使わない（NotebookLM送信や取得開始と誤解されるため）。

## 3. やらないことリスト（v1）

- 1記事1ファイル出力

- 抽出本文の編集機能
- リンクの自動巡回・クローリング（sitemap展開は1回の取得で終わる静的リストなので対象外）

- ログイン・クラウド同期
- 形態素解析などの高度な日本語検索
- PWA化・共有シート対応
- 100件超のパックの性能保証
- short_body以外の警告種別の追加（warnings配列は拡張可能だが、v1ではshort_bodyのみ）
- Worker側での本文抽出（HTMLRewriter移植はしない。抽出はブラウザ側の実績あるロジックを使う）

## 4. 画面構成

下部固定タブ3つ: **さがす｜候補（件数バッジ）｜仕上げ**

### 4.1 さがすタブ

- 上段: キーワード入力欄＋［探す］
- その下: 折りたたみ行「▸ 取り込み済みURL ◯件」。開くとURL一覧（個別削除可）＋URL貼り付け欄＋［URLを取り込む］
- 主役: 関連度カードの縦並び（スコア順）

**取り込みフロー（通常URL）**

1. URL貼り付け→［URLを取り込む］（改行・空白区切りで複数可。前後の空白は自動除去）
2. 複数件の場合: 「◯件のURLを取り込みました。本文を取得しますか？」→［本文を取得する］
3. 1件のみの場合: 即時取得してよい
4. 取得の同時実行数: 同一ホストへは同時1、別ホストなら合計2まで（定数で調整可）

**sitemap取り込み（最小形）**

- 貼り付けられたURLがsitemapと判定された場合（URL末尾が `.xml`、または取得結果が `urlset`/`sitemapindex` 形式）、展開モードになる
- XMLの取得はWorker経由、解析はブラウザ側で行う
- 展開結果は「◯件のURLが見つかりました」と件数表示し、チェックボックス付き一覧（全選択あり）から選んで［URLを取り込む］
- 展開上限200件。超過分は読み込まず「200件で打ち切りました」と警告する
- sitemapindex（入れ子）の場合は子sitemapのURL一覧を表示し、選んだ1つを展開する。多段の入れ子は非対応
- 取り込み後は通常URLと同じフロー（確認→取得）に乗る

**関連度カード**

- 表示: 関連度バッジ（高/中/低）／タイトル／ドメイン／約◯字／ヒット箇所の抜粋1〜2行（キーワード強調）
- ［立ち読みを開く］⇄［立ち読みを閉じる］。閉じるボタンは展開枠の上下両方に置く。冒頭500〜800字を高さ固定＋内部スクロールで表示。閉じたらそのカードの先頭へスクロールを戻す
- 未追加: ［パック候補に追加］
- 追加済み: 「✅ パック候補に追加済み」（押せないラベル）＋小さめの［候補から外す］。同一位置のボタントグルは誤タップを誘発するため禁止
- 取得失敗カード: 追加ボタンを出さない。失敗理由の文言＋［再試行］を表示
- 警告付きカード（short_body）: タイトル下に「⚠️ 本文が短いため要確認（約◯字）」を表示。立ち読み・候補追加は通常通り可能で、［最新版を取り直す］をカード面に出す
- 通常のfetchedカードでは［最新版を取り直す］はカード面に出さず、立ち読み展開内の下部に小さく置く
- 長いタイトル・URLは省略表示（…）

### 4.2 候補タブ

- パック内記事の一覧（タイトル＋ドメイン＋［候補から外す］）。並び順は追加順
- short_body警告付きの記事は一覧でも「⚠️ 約◯字」を添えて表示する
- 「◯件選択中」を表示。51件以上で警告バナー「50件を超えています。NotebookLMで扱いやすい量を超えている可能性があります」
- 下部に［仕上げへ進む］

### 4.3 仕上げタブ

- 選択件数の確認表示
- パック名の入力欄（既定: 「資料パック」。ファイル内の見出しとファイル名の両方に使う）
- ［パックのファイルを作る］: 保存済み本文からMarkdownを生成（通信なし）
- チェックボックス「テキスト版（.txt）も一緒に作る」（デフォルトOFF）
- 生成完了後: ［ファイルを保存］（Blobダウンロード）
- 注記文を常設: 「保存後、NotebookLMへはご自身でアップロードしてください」
- iOS向け案内を常設: 「iPhoneでは長期間使わないと保存データが消えることがあります。ホーム画面に追加すると防げます」

### 4.4 ボタン名一覧（文言を変更しないこと）

URLを取り込む／本文を取得する／探す／立ち読みを開く／立ち読みを閉じる／パック候補に追加／✅ パック候補に追加済み／候補から外す／再試行／最新版を取り直す／仕上げへ進む／パックのファイルを作る／ファイルを保存

注: 警告付きカードの再取得も「最新版を取り直す」に統一する。「再取得する」という別文言は作らない。sitemap展開後の取り込みも「URLを取り込む」を使う。

## 5. データモデル

articleとpackを分離する。さらにarticleはメタと本文をストア分離する。

### articleMeta

| フィールド | 型 | 説明 |
|---|---|---|
| id | string | 正規化URL（主キー） |
| originalUrl | string | 入力された元URL |
| finalUrl | string? | リダイレクト後URL（Workerが返す。記録のみ、重複判定には使わない） |
| title | string | 抽出タイトル |
| domain | string | 表示用ドメイン |
| charCount | number | 本文文字数 |
| fetchedAt | string? | 取得日時（ISO 8601） |
| fetchState | string | idle / fetching / fetched / failed |
| failReason | string? | network / http_error / extract_empty / too_large |
| warnings | string[] | 警告の配列。既定は空配列。v1で使う値は "short_body"（本文200字未満）のみ |
| httpStatus | number? | HTTPステータス（失敗時の表示用） |
| createdAt / updatedAt | string | ISO 8601 |

### articleBody

| フィールド | 型 | 説明 |
|---|---|---|
| id | string | articleMetaと同一キー |
| body | string | 抽出済み本文（プレーンテキスト、UTF-8） |

### pack

| フィールド | 型 | 説明 |
|---|---|---|
| id | string | v1では "default" 固定 |
| name | string | パック名（既定: 「資料パック」。仕上げタブで編集可） |
| items | string[] | 記事IDの配列。追加順＝出力順 |
| createdAt / updatedAt | string | ISO 8601 |

**設計原則**

- 候補状態をarticle側にフラグで持たせない。pack.itemsが唯一の真実
- scorerのスコアはDBに保存しない（検索時の一時値）
- 読み込み時に未知フィールドがあっても無視して壊れないこと（前方互換）
- 記事をURL一覧から削除した場合、articleMeta・articleBody・pack.items内の参照をすべて除去する
- IndexedDBには抽出後のプレーンテキストのみ保存し、生HTMLは保存しない

## 6. IndexedDB設計

- DB名: `notebooklm-packer`、バージョン: 1（schemaVersionはDBバージョンで管理）
- オブジェクトストア: `articleMeta`（keyPath: id）／`articleBody`（keyPath: id）／`pack`（keyPath: id）
- インデックス: articleMeta に fetchState、createdAt

**書き込み規約**

- 取得開始時: fetchState を fetching に更新してから取得を開始する
- 取得成功時（抽出完了後）: articleMetaとarticleBodyを**同一トランザクション**で書く
- 一覧表示はarticleMetaのみ読む。bodyは立ち読み・検索・生成時にのみ読む

**孤児レコード回収**

アプリ起動時、fetchState=fetching のまま残っているレコードをすべて failed（failReason=network）に倒す。

**マイグレーション**

- onupgradeneeded でバージョン分岐する構造を初版から用意する（v1→v2の分岐雛形をコメントで残す）
- 既存データの削除を伴うマイグレーションは行わない方針

**容量見積もり**: 1記事数十KB×50件＝数MB程度。v1では自動掃除は持たない。

**iOSの保存期限**: iOS Safari は7日間サイトを操作しないとIndexedDBを削除することがある（ITP）。仕様としては許容し、4.3のユーザー向け案内（ホーム画面に追加）でリスクを軽減する。

## 7. URL正規化ルールと重複ガード

方式: **ブラックリスト方式**。「消しすぎて別記事を同一視する」事故の方が「重複を見逃す」より重大、という前提で設計する。

**正規化手順（この順で適用）**

1. 前後の空白・改行を除去
2. スキームとホストを小文字化（パス・クエリは触らない）
3. デフォルトポートを除去（:80, :443）
4. フラグメント（#以降）を除去
5. 追跡パラメータのみ除去: `utm_*`, `fbclid`, `gclid`, `yclid`, `igshid`, `mc_cid`, `mc_eid`（定数リストで管理）
6. 残ったクエリパラメータはキー名でソートして保持
7. パスが「/」のみの場合は末尾スラッシュを除去。それ以外の末尾スラッシュは**保持**
8. httpからhttpsへの昇格はしない

**やらないこと**: AMP URL・モバイルURL（m.〜）の同一視。finalUrlベースの重複検知（将来検討）。

**重複ガード**: 正規化URLの完全一致のみ。取り込み時に一致したら新規作成せず、既存記事がある旨を表示する。sitemap展開からの取り込みでも同一の重複ガードを通す。

**正規化テストケース（実装時にユニットテストとして固定すること）**

| # | 入力 | 期待結果 |
|---|---|---|
| 1 | `HTTPS://Example.COM/Page` | `https://example.com/Page` |
| 2 | `https://example.com/a?utm_source=x&id=3` | `https://example.com/a?id=3` |
| 3 | `https://example.com/a?page=2` | そのまま保持 |
| 4 | `https://example.com/a#section` | `https://example.com/a` |
| 5 | `https://example.com/` | `https://example.com` |
| 6 | `https://example.com/a/`（記事パス） | 末尾スラッシュ保持 |
| 7 | `https://example.com:443/a` | `https://example.com/a` |
| 8 | `http://example.com/a` | httpのまま |
| 9 | `https://example.com/a?b=2&a=1` | `https://example.com/a?a=1&b=2` |
| 10 | ` https://example.com/a `（前後空白） | 空白除去 |
| 11 | `https://example.com/a?fbclid=xyz` | `https://example.com/a` |
| 12 | `https://m.example.com/a` | そのまま（PC版と同一視しない） |
| 13 | 1と同URLの再投入 | 重複として弾かれる |

## 8. 状態遷移

### article.fetchState

| 現在 | イベント | 遷移先 | 備考 |
|---|---|---|---|
| （なし） | URL取り込み | idle | articleMetaのみ作成 |
| idle | 取得開始 | fetching | |
| fetching | 取得＋抽出成功 | fetched | meta＋bodyを同一トランザクションで保存 |
| fetching | 取得失敗または抽出失敗 | failed | failReasonを記録 |
| fetching | アプリ再起動 | failed（network） | 起動時の孤児回収 |
| failed | ［再試行］ | fetching | |
| fetched | ［最新版を取り直す］ | fetching（再取得モード） | 下記参照 |

**short_body警告**: 抽出成功時にcharCountが200字未満だった場合、fetchStateはfetchedのまま warnings に "short_body" を付与する。failedにはしない。本文は通常通り保存し、立ち読み・候補追加・取り直しがすべてできる。取り直しに成功してcharCountが200字以上になったら警告を外す。

**再取得モードの重要仕様**: 再取得が失敗した場合、既存の本文・メタを**保持したまま** fetchState を fetched に戻し、エラーはトースト等で通知するのみ。再取得失敗で保存済みデータを失ってはならない。置き換えは成功時のみ。

### 候補状態（pack.itemsへの所属で表現）

| 現在 | イベント | 結果 |
|---|---|---|
| 非所属（fetchedのみ） | ［パック候補に追加］ | itemsに追加。重複追加は不可 |
| 所属 | ［候補から外す］ | itemsから除去。記事データは残す |
| 所属 | 記事をURL一覧から削除 | itemsからも除去 |

追加可能条件は fetchState=fetched のみ。idle／fetching／failed の記事には追加ボタンを表示しない。short_body警告付きの記事はfetchedなので追加できる。

### 仕上げ

Markdown/txt生成は通信なしのローカル処理（IndexedDBからの読み出しは非同期だが、ネットワークは使わない）。items順に1件ずつ連結する。途中でbodyが見つからない記事はスキップし、生成結果の末尾に「◯件をスキップしました」と注記する。

## 9. エラー分類・警告分類とWorker応答仕様

### エラー分類（fetchState=failed になるもの）

| failReason | 判定場所 | 判定 | カード表示文言 | アクション |
|---|---|---|---|---|
| network | Worker/ブラウザ | fetch失敗・タイムアウト | 接続できませんでした | ［再試行］ |
| http_error | Worker | HTTP 4xx/5xx | ページを取得できませんでした（{status}） | ［再試行］＋URL確認を促す補足 |
| too_large（一次） | Worker | 生HTMLが5MB超 | ページが大きすぎるため対象外です | なし |
| extract_empty | ブラウザ | 抽出本文が空（空白のみ含む） | 本文を取り出せませんでした（会員限定ページの可能性があります） | ［再試行］ |
| too_large（二次） | ブラウザ | 抽出本文が50万字超 | ページが大きすぎるため対象外です | なし |

too_largeは二段防御とする。Worker側はバイトサイズ（抽出前にメモリを守る防壁）、ブラウザ側は抽出後の文字数。ユーザーから見た表示は同一でよい。

### 警告分類（fetchState=fetched のまま。失敗ではない）

| warning | 判定場所 | 判定 | カード表示 | 選べる操作 |
|---|---|---|---|---|
| short_body | ブラウザ | charCountが200字未満 | ⚠️ 本文が短いため要確認（約◯字） | 立ち読みを開く／最新版を取り直す／パック候補に追加 |

short_bodyを失敗ではなく警告にする理由: 短いページにも資料価値がある場合があり、捨てる判断は人間が行う。一方too_largeを失敗のままにするのは、巨大本文の保存・表示に実コストがあるため。

閾値（short_body 200字／本文too_large 50万字／生HTML上限5MB）とタイムアウト（Worker側15秒）は定数で管理し調整可能にする。extract_empty・too_large（二次）・short_bodyの判定がすべてブラウザ側に揃うため、閾値管理は一箇所にまとめる。

### Worker応答仕様（JSON）

- 成功: `{ ok: true, html, finalUrl, httpStatus }`
- 失敗: `{ ok: false, reason: "network" | "http_error" | "too_large", httpStatus? }`

Worker側の要件:

- 文字コードを判定しUTF-8に正規化したHTMLを返す（Shift_JIS等のサイト対応。`response.text()`で直接復号しない）
- finalUrlはリダイレクト追従後の `response.url` を返す（要求URLをそのまま返さない）
- 生HTMLのバイトサイズ上限（5MB）とタイムアウト（15秒）をWorker側で適用する
- 本文抽出はWorker側では行わない（ブラウザ側の既存ロジック `prepareDoc` / `nodeToPlainText` 系を流用する）
- sitemap XMLの取得も同じWorker経由で行う。XML解析はブラウザ側

### 旧Workerからの改修ポイント（audit.md対応）

- 対象サイトが4xx/5xxでもstatus 200＋`X-Source-Status`ヘッダで返す方式を廃止し、上記JSON応答に統一する
- 新Workerは別名でデプロイし、旧Workerは現行アプリ（`v1/`）用に残す

## 10. 検索とscorer差し込み口

### 検索仕様（v1.2）

- 対象: fetchState=fetched の全記事の本文＋タイトル（short_body警告付きも含む）
- クエリ: スペース区切りで複数語。いずれかの語を含む記事をすべて拾い（OR）、含む語の種類が多い記事ほど上位に表示する。全語を含む記事が最上位帯に来る
- 表記ゆれ対応: 「醤油 しょうゆ」のように同義の表記を並べて入力する使い方を想定する。構文記号（｜や - など）は導入しない
- 大文字小文字は同一視
- 空クエリ時: 全記事を取り込みの新しい順に表示
- 関連度バッジ: スコアを高/中/低の3段階に割ってカードに表示する（閾値は定数で調整可。導出はUI側で行う）

### scorerインターフェース（唯一のAI差し込み口）

- 入力: 検索クエリ（string）、記事の配列（メタ＋本文）
- 出力: 記事ごとの `{ score: number, highlights: ヒット位置の配列 }`
- v1実装: 語ごとのヒット回数＋含有語種数ボーナス＋タイトル一致ボーナスでスコア化
- 高/中/低ラベルはscorerの責務に含めない（UI側でスコア帯から導出）
- スコアはDBに保存しない
- 将来GemmaやWorkers AIに差し替える際は、この関数の**中身だけ**を置き換える。入出力の形は変えない
- 除外キーワード導入時（v1.3）は、入力に除外語リストを追加する形で拡張する
- 要約・自動タグなど他の差し込み口は予約しない

### Phase 11の「⚠️注意」表示について

実装フェーズの冒頭で、現行アプリ（Phase 11）の⚠️表示が何を意味していたかをコードから確認すること。本仕様のshort_body警告と意味が重複する場合は統合する。独自の意味を持つ場合は実装せず、内容を報告して相談する。

## 11. 出力仕様

- 正本: Markdown、パック全体で1ファイル
- 任意: 同内容のtxt版（記号を除いたプレーン構造）
- ファイル名: `{パック名}_YYYYMMDD_HHmm.md`（txtも同名）。パック名はファイル名に使えない文字（`/ \ : * ? " < > |` 等）を `_` に置換する
- 保存: Blobダウンロード

**ファイル構造**

```
# {パック名}

- 作成日時: 2026-06-12 14:30
- 記事数: 12件

## 目次

1. 記事タイトルA（example.com）
2. 記事タイトルB（example.org）

---

## 1. 記事タイトルA

- 元URL: https://example.com/article
- ドメイン: example.com
- 取得日時: 2026-06-10 09:12
- 文字数: 約4,200字

（本文プレーンテキスト）

---

## 2. 記事タイトルB
（以下同様）
```

各記事の冒頭に元URLを必ず焼き込む（NotebookLM上で出典を辿れるようにするため）。本文はプレーンテキストとしてそのまま埋め込む。short_body警告はパック出力には記載しない（人間が選んだ時点で資料として確定とみなす）。

## 12. 受け入れ条件

1. 複数URLを貼って取り込み→確認1枚→取得→カードがスコア順に並ぶ
2. カードで立ち読みの開閉ができ、閉じるとカード先頭へスクロールが戻る
3. fetchedの記事にのみ追加ボタンが出て、追加すると✅ラベル＋バッジが+1される
4. 同じURL（正規化バリエーション込み）を2回取り込んでも記事は1件のまま
5. 候補タブで外せて、外しても記事データと立ち読みは生きている
6. 仕上げでmdが生成され、先頭にパック概要・目次、各記事にメタヘッダが付く
7. ブラウザを完全に終了して再起動しても、カゴと本文が残っている（iOSの7日削除は仕様として許容し、案内文で軽減する）
8. 取得中にリロードしても、再起動後にfailedとして表示され再試行できる
9. 「送信」という語がUI上のどこにも存在しない
10. 30件のパックで一覧・検索・生成が体感ストレスなく動く。51件で警告が出る
11. 本文200字未満のページが警告付きで保存され、立ち読み・候補追加・取り直しができる
12. sitemap URLを貼ると展開一覧が出て、選んだURLだけが取り込まれる。201件以上のsitemapは200件で打ち切り警告が出る
13. 「醤油 しょうゆ」で検索すると、片方しか含まない記事も一覧に出て、両方含む記事が上位に表示される
14. カードに関連度バッジ（高/中/低）が表示される
15. 404を返すURLが「取得失敗（404）」として表示される（成功扱いにならない）
16. Shift_JISのサイトが文字化けせずに取得・表示される

## 13. テスト観点

- URL正規化テスト13本（7章の表）がすべて通る
- utm付きURLと素URLが同一記事になる／`?page=2` は別記事になる
- 404／タイムアウト／生HTML 5MB超／抽出本文空／本文50万字超が、それぞれ正しい分類で表示される
- 150字程度のページがshort_body警告付きのfetchedになり、立ち読みと候補追加ができる
- short_bodyの記事で［最新版を取り直す］が失敗しても、元の短い本文が残っている
- 取り直しで200字以上になったらshort_body警告が外れる
- Shift_JISのサイトで文字化けしない
- リダイレクトするURLでfinalUrlにリダイレクト先が記録される
- sitemap: 正常展開／201件以上の打ち切り警告／sitemapでないXML／sitemapindexの子一覧表示
- OR検索: 「醤油 しょうゆ」で両方含む記事が片方のみの記事より上位に来る
- 関連度バッジが高/中/低の3段階で表示される
- 取得中のリロード→起動時に孤児がfailedへ回収される
- オフライン中: 取得は失敗するが、立ち読み・候補操作・仕上げは動く
- 追加済みラベルを連打しても候補から外れない（誤タップ耐性）
- 候補50件投入で警告が出て、生成も成功する
- パック名に「/」等を含めた場合、ファイル名が正しくサニタイズされる
- iOS SafariとAndroid Chromeの実機でファイル保存ができ、NotebookLMにアップロードできる

## 付録: 実装フェーズ計画
