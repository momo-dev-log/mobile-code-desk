NotebookLM資料ツール 実装着手指示書 v1.0

最終更新: 2026-06-13

この指示書は、仕様書 "docs/notebooklm-tool-spec-v1.0-text.md" を唯一の基準として、"notebooklm-packer" の実装に入るための作業指示である。

仕様書とこの指示書が食い違う場合は、仕様書を優先する。

現存する "docs/spec.md" や "docs/audit.md" は、今回の実装基準として使わない。

---

0. 大前提

- 仕様書 v1.0 を唯一の基準とする
- 仕様書に記載のない機能は実装しない
- 仕様書「外す機能」「やらないことリスト」にある機能は、良い案に見えても実装しない
- 思いついた追加案は、実装せず報告だけ行う
- 一気に全実装しない
- 下記のフェーズ単位で進める
- 各フェーズ完了ごとに、動作確認できる状態にして報告する
- UI文言に「問い」「送信」を使わない
- 出発点は現行v2の "notebooklm-packer/"
- v1へ戻さない
- 既存IndexedDBデータは引き継がない
- 新スキーマで作り直す
- テスト記事・旧仕様データは消えてよい

---

1. 着手前に必ず確認・報告すること

コードを書く前に、次の3点を実際のコードベースとgit履歴で確認し、報告すること。

この確認が済むまで、実装に入らない。

1.1 現在の作業ブランチと対象ディレクトリ

確認すること:

- 現在の作業ブランチ
- どのブランチの "notebooklm-packer/" を改修するのか
- 新ブランチを切るべきか
- 前回 "claude/notebookml-resource-pack-feasibility-aPfjF" で作業していた認識が正しいか
- "notebooklm-packer/v1/" に旧版を退避していた認識が正しいか

1.2 流用する中核ファイルの現状

確認すること:

- 本文取得はどのファイルに実装されているか
- Worker呼び出しはどのファイルに実装されているか
- 抽出ロジックはどのファイルに実装されているか
- IndexedDB層はどのファイルに実装されているか
- Markdown生成はどのファイルに実装されているか
- 複数URL取り込みUIはどのファイルに実装されているか
- Markdown生成の文字化け対策がどう入っているか
  - Blobのcharset指定
  - BOM付与
  - その他、日本語が文字化けしないための処理

文字化け対策は必須品質として引き継ぐ。

1.3 変更対象ファイルの推定と実物の一致

この指示書の「変更対象ファイル」は推定である。

実物と照合し、ズレがあれば報告すること。

以下のフェーズ順で進める。

フェーズ1: データ層

目的

IndexedDBを仕様書8章の形に作り替える。

UIはまだ作らない。
このフェーズでは、後続フェーズが使う保存・取得・削除・カテゴリ操作の土台を作る。

やること

- IndexedDBを3ストア構成にする
  - "articleMeta"
  - "articleBody"
  - "category"
- "articleMeta" に以下を持たせる
  - "normalizedUrl"
  - "originalUrl"
  - "title"
  - "domain"
  - "status"
  - "categoryId"
  - "isExported"
  - "fetchedAt"
- "status" は "success" / "failed" の2値にする
- "categoryId" は未分類の場合は空にする
- "isExported" の初期値は "false"
- 取得成功記事は "articleMeta" と "articleBody" の両方を持つ
- 取得失敗URLは "articleMeta" のみを持つ
- 取得失敗URLは "articleBody" を持たない
- 既存v2のIndexedDBデータは引き継がない
- DBバージョンを上げ、新スキーマで作り直す

重複ガード

取り込み時、正規化URLが "articleMeta" に存在すれば弾く。

成功記録でも失敗記録でも、"articleMeta" に記録があれば重複作成しない。

削除済みで "articleMeta" に記録がないURLは、再取り込みできる。

カテゴリ削除時の戻し処理

カテゴリ削除時、そのカテゴリに属する記事の "categoryId" を空に戻す。

記事自体は削除しない。

基本操作

後続フェーズで使うため、以下の操作を用意する。

- 記事保存
  - meta + body
- 取得失敗記録保存
  - metaのみ
- 記事一覧取得
- 記事単体取得
- 記事削除
  - "articleMeta" と "articleBody" の両方を削除
- 取得失敗URL削除
  - "articleMeta" を削除
- カテゴリ作成
- カテゴリ名変更
- カテゴリ削除
- カテゴリ一覧取得
- 記事へのカテゴリ割り当て
- "isExported" を立てる操作

カテゴリ名の空欄不可・同名不可チェックは、この段階でデータ層側にも土台を用意する。

確認方法

一時的な確認用UIは原則作らない。

仮ボタンや仮表示が残ると、仕様外UIが混ざる危険があるためである。

確認は、実装側の内部確認・ログ・IndexedDB確認・テストで行う。
どうしても確認用UIが必要な場合は、作る前に相談する。

このフェーズで土台として関係する受け入れ条件

- 条件10: 記事を削除すると、同じURLを再度取り込める
- 条件11: 既に "articleMeta" に残っているURLは重複作成されない
- 条件12: カテゴリを削除すると、そのカテゴリの記事は未分類に戻り、記事自体は消えない
- 条件15: 再起動後も、記事・カテゴリ・出力済み状態が残る

フェーズ5: 取得失敗エリア

目的

取得失敗URLを、成功記事リストとは別枠で扱う。

やること

- 取得失敗が1件以上ある時だけ、記事タブ上部に取得失敗エリアを表示する
- 0件の時は表示しない
- 赤文字で「取得失敗 N件」と分かるようにする
- 各失敗URLに以下を置く
  - 再試行
  - 開く
  - 削除
- 取得失敗エリアはカテゴリバーの絞り込み対象にしない
- 「すべて」「未分類」「各カテゴリ」のどれを選んでいても、取得失敗が1件以上あれば記事タブ上部に表示する
- 失敗URLは "articleMeta" に記録する
- 失敗URLは "articleBody" を持たない
- 失敗URLはカテゴリ付け対象にしない
- 失敗URLはMarkdown出力対象にしない
- 同じ失敗URLを重複表示しない

取得失敗の定義

取得失敗とは、記事素材として保存できなかったものを指す。

対象:

- URL取得ができなかった
- HTML取得ができなかった
- 本文抽出ができなかった

本文が短いだけのものは取得失敗扱いにしない。

失敗理由の細分類はしない。
"success" / "failed" の2値で扱う。

このフェーズで関係する受け入れ条件

- 条件6: 取得失敗が1件以上ある時だけ、記事タブ上部に取得失敗エリアが表示される
- 条件7: 取得失敗エリアはカテゴリバーの絞り込み対象にならない
- 条件8: 取得失敗URLには［再試行］［開く］［削除］がある
- 条件9: 本文が短いだけの記事は取得失敗扱いにならない
- 条件11: 既に "articleMeta" に残っているURLは重複作成されない

フェーズ10: Markdown出力

目的

カテゴリ単位でMarkdownを書き出せるようにする。

やること

- カテゴリ単位でMarkdownを生成する
- 全体まとめ出力は作らない
- 未分類のMarkdown出力は基本的に作らない
- ファイル名は "カテゴリ名_YYYY-MM-DD.md"
- ファイル名に使えない文字は "_" に置換する
- 各記事の冒頭に元URLを必ず入れる
- 本文はプレーンテキストで出力する
- 装飾しない
- 出力に成功した記事に "isExported" を立てる
- UI上では赤い印と「出力済み」を表示する
- 出力済みの記事も一覧に残す
- 出力済みの記事も再出力できる
- 再出力はカテゴリ内の全記事を対象にする
- 特別な再出力ボタンは作らない
- Markdown出力後に「削除しますか？」の確認は出さない

出力構造

# カテゴリ名

作成日: YYYY-MM-DD
記事数: N

---

## 記事タイトル

- URL: 元URL
- ドメイン: example.com

本文

---

文字化け対策

日本語が文字化けしないことは必須品質である。

Shift_JISサイトも、取得後はUTF-8本文として保存し、Markdown出力時にも文字化けしないこと。

既存v2でBlobのcharset指定やBOM付与などの対策が入っている場合、必ず引き継ぐ。

このフェーズで関係する受け入れ条件

- 条件13: Shift_JIS由来の本文も、Markdown出力時に日本語が文字化けしない
- 条件15: 再起動後も、記事・カテゴリ・出力済み状態が残る

5. 確定済みの設計判断

迷った場合は、以下に従う。

- 画面は記事タブ／カテゴリタブの2タブ
- 1画面集約にしない
- 1記事1カテゴリ
- 複数カテゴリ重複所属はしない
- 未分類は出力対象ではない
- 出力はカテゴリ単位のみ
- 全体まとめ出力はしない
- 出力と削除は別操作
- 出力後の削除確認は出さない
- 出力後も記事は残す
- 出力済み記事も再出力できる
- カテゴリは削除しない限り常駐
- カテゴリ削除時、中の記事は未分類に戻る
- カテゴリ名は空欄不可・同名不可
- 取り込みは「取り込む」で即取得
- 取得中はURLごとに取得中カードを表示する
- 重複URLは、"articleMeta" に記録があれば弾く
- 成功記録でも失敗記録でも、重複作成しない
- 削除済みで "articleMeta" に記録がなければ再取り込みできる
- 取得失敗エリアは、取得失敗が1件以上ある時だけ表示する
- 取得失敗エリアは記事タブ上部に独立表示する
- 取得失敗エリアはカテゴリバーの絞り込み対象にしない
- 本文が短いだけのものは取得失敗扱いにしない
- 本文短め警告は出さない
- 失敗理由の細分類はしない
- 検索は初版で実装しない
- サイトマップ展開は実装しない
- サイトマップ自動探索は実装しない
- NotebookLM自動送信はしない
- iOS Safariの保存期限への特別対応はしない
- IndexedDBで進める
- 保存保証の警告文や追加対策は不要
- Markdown出力の日本語文字化けゼロは必須品質

---

6. 進め方のルール

- フェーズごとに区切って実装する
- 各フェーズ実装前に、作業計画を出す
- 承認を得てから、そのフェーズに着手する
- フェーズ完了ごとに報告する
- 報告には、満たした受け入れ条件番号を添える
- 各フェーズで動作確認できる状態を作る
- 仕様書にない機能は実装しない
- やらないことリストの機能は実装しない
- 提案がある場合は、実装せず報告する
- 一時的な確認用UIは原則作らない
- 必要な場合は、事前に相談する
- まずフェーズ1の実装計画を提示し、承認を得てから着手する

---

7. 完了報告の形式

フェーズ完了ごとに、以下の形で報告する。

## フェーズX 完了報告

### 実装した内容

### 変更したファイル

### 満たした受け入れ条件

### テスト結果

### 未確認の項目

### 次に進んでよいか

仕様書にない機能を追加した場合は、完了扱いにしない。

不要機能を復活させた場合も、完了扱いにしない。
